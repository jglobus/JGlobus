GridShib SAML Tools

This is the install document for GridShib SAML Tools. See the readme document for more information about GridShib, the GridShib SAML Tools, and other GridShib software components.

GridShib SAML Tools is a standalone software package. The only requirements for installation are Java 1.4.2 (or later) and Ant 1.6.2 (or later). The popular tool openssl is useful (for inspecting the contents of certificates) but is not required.

GridShib is licensed under the Apache License, Version 2.0. Please review the licensing terms of the GridShib License before installing this software. Additional licensing information will be found on the GridShib Downloads page.

Getting Started

GridShib SAML Tools was previously available here:

  • http://gridshib.globus.org/download.html#saml-tools

But the most recent version maintained for TeraGrid Science Gateway use is here:

  • https://sourceforge.net/projects/cilogon/files

Choose your preferred download format (ZIP or GZIP), download the distribution file, and continue with the Installation and Testing sections. As necessary, replace X_X_X in any examples with the version you're using.

Installation

Note: When installing the GridShib SAML Tools, choose an installation directory that does not have any spaces in its path name; otherwise the commands below will fail.

To install the GridShib SAML Tools, download the distribution file into a working directory (e.g., /tmp/gridshib-saml-tools-X_X_X-src.tar.gz) and perform the following two-step installation process:

  1. Unpack the GridShib SAML Tools distribution file. For example, on UNIX systems type:

      UNIX:
  $ cd /usr/local
  $ gunzip /tmp/gridshib-saml-tools-X_X_X-src.tar.gz
  $ tar xvf /tmp/gridshib-saml-tools-X_X_X-src.tar

    In the case of Windows, simply double-click the ZIP archive to install the distribution. In what follows, we assume the GridShib SAML Tools are installed in directory c:\gridshib on Windows systems.

  2. Run 'ant install' on the GridShib SAML Tools build file. This ant target creates some JAR files and runs a test script. For example, on UNIX or Windows systems (resp.) type:

      UNIX:
  $ cd /usr/local/gridshib-saml-tools-X_X_X
  $ ant install
      Windows:
  > cd c:\gridshib\gridshib-saml-tools-X_X_X
  > ant install

As a result of step 2 above, the following operations are performed by the install script:

  1. A bootstrap properties file is written to the src/org/globus/gridshib/resource directory.
  2. A log4j properties file is written to the etc directory.
  3. Three JAR files (gridshib-saml-tools-X_X_X.jar, gridshib-common-X_X_X.jar, and globus-opensaml-1.1.jar) are created and written to the lib directory.
  4. A logs directory is created and an empty log file (saml-tools.log) is created.
  5. A test script is run and a default holder-of-key SAML assertion is written to the console (test #1).
  6. A test script is run and an explicit (but identical) holder-of-key SAML assertion is written to the console (test #2).
  7. A test script issues a sender-vouches SAML assertion, which is also written to the console (test #3).
  8. The test script (test #4) issues another SAML assertion and writes it to file testassertion.xml in the home directory.
  9. The test script (test #5) binds the SAML assertion issued at the previous step to an X.509 proxy certificate and writes the certificate to file testproxy.pem in the home directory.
  10. The test script (test #6) issues another SAML assertion, encodes it as a DER-encoded ASN.1 structure and writes this structure to file testextension.der in the home directory.
  11. The test script (test #7) binds the DER-encoded ASN.1 structure created at the previous step to an X.509 proxy certificate and writes the certificate to file testproxy2.pem in the home directory.
  12. The test script (test #8) issues yet another SAML assertion, binds it to an X.509 proxy certificate, and writes the certificate to file testproxy3.pem in the home directory.
  13. The test script (test #9) issues a SAML assertion similar to one issued by a TeraGrid Science Gateway. The script binds the assertion to an X.509 proxy certificate and writes the certificate to file testgatewayproxy.pem in the home directory.

Installation of the GridShib SAML Tools is now complete.

Testing

To use any of the command-line scripts, an environment variable is required:

  UNIX:
  $ export GRIDSHIB_HOME=/usr/local/gridshib-saml-tools-X_X_X
  Windows:
  > set GRIDSHIB_HOME=c:\gridshib\gridshib-saml-tools-X_X_X

As you can see, the GRIDSHIB_HOME environment variable is set to the absolute path of the installation directory.

We are now ready to start testing. As indicated in the previous section, the installer runs a test script that issues assertions and certificates. Before proceeding, make sure the test files (testassertion.xml, testproxy.pem, testextension.der, testproxy2.pem, and testproxy3.pem) were actually created in the GridShib home directory. The SAML assertion (testassertion.xml) is an ordinary text file that can be read with any text editor. The proxy certificates can be viewed with openssl commands such as the following:

  UNIX:
  $ openssl x509 -text -in $GRIDSHIB_HOME/testproxy.pem
  $ openssl x509 -text -noout -in $GRIDSHIB_HOME/testproxy2.pem
  $ openssl x509 -text -noout -certopt ext_parse -in $GRIDSHIB_HOME/testproxy3.pem

Logging and Error Handling

By default, log output is written to file saml-tools.log in the logs directory:

  UNIX:
  $ more $GRIDSHIB_HOME/logs/saml-tools.log
  Windows:
  > more < %GRIDSHIB_HOME%\logs\saml-tools.log

Also, the default log level is set to INFO in the log configuration file:

  UNIX:
  $ more $GRIDSHIB_HOME/etc/log4j.properties
  Windows:
  > more < %GRIDSHIB_HOME%\etc\log4j.properties

You can change the log configuration by modifying the log4j.properties file. For example, if the GridShib SAML Tools will be used by multiple users, you may want to log to individual home directories by setting the following property in the log4j.properties file:

  log4j.appender.R.File=${user.home}/logs/saml-tools.log

Also, you can either change the log level in the configuration file or set the log level to DEBUG on-the-fly:

  UNIX:
  $ $GRIDSHIB_HOME/bin/gridshib-saml-extract --debug \
      --infile $GRIDSHIB_HOME/testproxy.pem
  Windows:
  > %GRIDSHIB_HOME%\bin\gridshib-saml-extract --debug
      --infile %GRIDSHIB_HOME%\testproxy.pem

The --debug command-line option may be added to any invocation of a GridShib tool or client. Doing so affects the verbosity of error messages written to the log file. If you set the log level to DEBUG in the log4j.properties file, the --debug command-line option has no effect.

In addition to logging, the scripts write short error messages to the terminal (stderr) by default. To run the script in quiet mode, add the --quiet option to any command-line invocation. In quiet mode, if the command fails, it will return a nonzero error code but it will not send any output to stderr. Instead, all output will be written to the log file.

Normally, the order of options on the command line is irrelevant. If there is a command-line parsing error, however, the parser stops parsing at the point the error occurs. Therefore, it's a good habit to put important options (such as --debug and --quiet) first, otherwise they may be ignored in the event an error occurs.

Configuration

This section shows how to configure the GridShib SAML Tools. The configuration options outlined here affect the use of the command-line tools only. Java API developers should consult the Developer Guide for configuration options.

Note: We recommend moving on to the User Guide and trying the examples before modifying the default bootstrap properties. If you do alter the configuration, be sure to re-install the SAML Tools as shown previously.

Users of the command-line tools have two levels of configuration. The low-level bootstrap properties are discussed in this section, while the high-level configuration properties for the SAML Tools are covered in the configuration section of the User Guide.

Bootstrap properties

The build may be configured by modifying the build.properties file in the home directory. In particular, the following bootstrap properties are configurable:

gridshib.logConfigPath (required)
Path to the log configuration file (default: etc/log4j.properties)
gridshib.SAMLToolsConfigPath (required)
Path to the configuration properties file for the GridShib SAML Tools (default: etc/gridshib-saml-tools-config.properties)
gridshib.certPath (required)
Path to the certificate of the default issuing credential (default: etc/testcert.pem)
gridshib.keyPath (required)
Path to the private key of the default issuing credential (default: etc/testkey.pem)
gridshib.identityAttributesPath (required)
Path to a file list of identity attributes, that is, attributes whose values are globally unique principal names (default: etc/identity-attributes.txt)
gridshib.cogConfigPath (required)
Path to the jglobus CoG configuration file (default: etc/cog.properties)
gridshib.SecurityContextImpl (optional)
The fully qualified classname of a custom implementation of the SecurityContext interface (default: none)

You can modify any of the paths in the build.properties file and then re-install the SAML Tools to persist a new bootstrap properties file. A path can be relative or absolute. A relative path is relative to the GridShib home directory ($GRIDSHIB_HOME or %GRIDSHIB_HOME%).

gridshib.logConfigPath

The gridshib.logConfigPath bootstrap property points to a log configuration properties file. To use some other log properties file, set the bootstrap property in the build.properties file. For example:

  gridshib.logConfigPath=etc/log4j-syslog.properties

A default log configuration properties file is hardcoded in the build file as $GRIDSHIB_HOME/etc/log4j.properties. Regardless of the value of the gridshib.logConfigPath bootstrap property, this default log properties file is dynamically created at install time.

For more information on logging, see the section on Logging and Error Handling.

gridshib.SAMLToolsConfigPath

The gridshib.SAMLToolsConfigPath bootstrap property points to the default configuration file for the command-line tools. See the User Guide for instructions how to override the default configuration file on the command line and how to write configuration files for the SAML Tools.

gridshib.certPath and gridshib.keyPath

The gridshib.certPath and gridshib.keyPath bootstrap properties together give the path to the default issuing credential. If no credential is configured in the configuration properties file (gridshib.SAMLToolsConfigPath), the software defaults to the credential in the bootstrap properties file, that is, the credential given by gridshib.certPath and gridshib.keyPath.

gridshib.identityAttributesPath

Attributes whose values are globally unique principal names are called identity attributes. For example, an e-mail address (FriendlyName="mail") is an identity attribute. Because identity attribute values are globally unique, they play a potentially important role in applications. In GridShib for GT, for instance, identity attribute values are blacklistable.

The gridshib.identityAttributesPath bootstrap property points to a file containing a list of identity attributes. The list is comprehensive, so it's unlikely you'll ever need to modify this file, let alone provide a replacement.

gridshib.cogConfigPath

The gridshib.cogConfigPath bootstrap property points to an empty jglobus CoG configuration file. This file exists mainly to avoid a nasty error message from appearing in the logs. If you find a use for this configuration file, please send e-mail to the gridshib-user mailing list describing your use case.

gridshib.SecurityContextImpl

For command-line use, the gridshib.SecurityContextImpl property should be omitted (as is the case in the build.properties file). This property is mainly of interest to developers. See the Developer Guide for more information.

Bootstrap properties file

The bootstrap properties defined in the build.properties file are written to an ordinary Java properties file at a well-known location on the classpath.

  UNIX:
  $ jar -xvf $GRIDSHIB_HOME/lib/gridshib-saml-tools-X_X_X.jar \
      org/globus/gridshib/resource/gridshib-bootstrap.properties
  $ cat org/globus/gridshib/resource/gridshib-bootstrap.properties
  Windows:
  > jar -xvf %GRIDSHIB_HOME%\lib\gridshib-saml-tools-X_X_X.jar
      org/globus/gridshib/resource/gridshib-bootstrap.properties
  > type org/globus/gridshib/resource/gridshib-bootstrap.properties

This default bootstrap properties file is automatically created when you install GridShib SAML Tools. The command-line tools use this properties file by default. Alternatively, you can specify a path to a separate bootstrap properties file that completely overrides the default bootstrap properties created by the install process. This can be done by setting the absolute path to the alternate bootstrap properties file in one of two ways:

  1. Set the GRIDSHIB_CONFIG environment variable.
  2. Specify the org.globus.gridshib.config system property on the ant command line using the -D option.

The org.globus.gridshib.config system property takes precedence, so if both the environment variable and the system property are set, the latter takes effect.

Note: As with GRIDSHIB_HOME, choose a path to the bootstrap properties file (GRIDSHIB_CONFIG) that does not have any spaces in the path name.
